# L2 - Detailed Requirements

## Cooperative Management Platform

This document defines detailed requirements with acceptance criteria. Each requirement traces to a parent L1 high-level requirement.

---

## User Account Management (L1-REQ-001)

### L2-REQ-001: Create User Account

**Traces to:** L1-REQ-001

The system shall allow creation of a new user account with a username and password.

**Acceptance Criteria:**

- AC1: A user is created with a unique username and a securely hashed password.
- AC2: The password is stored as a salted hash using a cryptographically secure random 16-byte salt.
- AC3: The system returns the created user's identifier upon success.
- AC4: The system rejects creation if the username already exists.

---

### L2-REQ-002: Update User Account

**Traces to:** L1-REQ-001

The system shall allow updating an existing user's details.

**Acceptance Criteria:**

- AC1: An authorized user can update username and other account properties.
- AC2: The system returns the updated user record upon success.

---

### L2-REQ-003: Soft Delete User Account

**Traces to:** L1-REQ-001

The system shall support soft deletion of user accounts, preserving data for audit purposes.

**Acceptance Criteria:**

- AC1: Deleting a user sets the `IsDeleted` flag to true rather than physically removing the record.
- AC2: Soft-deleted users cannot authenticate.
- AC3: The system returns confirmation of deletion.

---

### L2-REQ-004: Change Password

**Traces to:** L1-REQ-001

The system shall allow users to change their own password by providing the current password.

**Acceptance Criteria:**

- AC1: The user must supply a valid current password to change their password.
- AC2: If the current password is incorrect, the request is rejected.
- AC3: The new password is stored with a new salted hash.

---

### L2-REQ-005: Admin Password Reset

**Traces to:** L1-REQ-001

The system shall allow administrators to reset a user's password without requiring the old password.

**Acceptance Criteria:**

- AC1: An administrator can set a new password for any user.
- AC2: The new password is stored with a new salted hash.

---

### L2-REQ-006: Check Username Availability

**Traces to:** L1-REQ-001

The system shall allow checking whether a username is already taken.

**Acceptance Criteria:**

- AC1: The endpoint is accessible without authentication.
- AC2: Returns a boolean indicating whether the username exists.

---

## Authentication and Authorization (L1-REQ-002)

### L2-REQ-007: User Authentication

**Traces to:** L1-REQ-002

The system shall authenticate users via username and password and return a JWT access token.

**Acceptance Criteria:**

- AC1: A POST request with valid username and password returns a JWT token and the user's identifier.
- AC2: The JWT token contains claims for UserId, Username, Roles, and Privileges.
- AC3: Invalid credentials return an appropriate error response.
- AC4: The authentication endpoint is accessible without prior authentication.

---

### L2-REQ-008: Resource-Based Authorization

**Traces to:** L1-REQ-002

The system shall enforce resource-level authorization on API endpoints using role privileges.

**Acceptance Criteria:**

- AC1: Each protected endpoint declares the required operation (Create, Read, Write, Delete) and target aggregate.
- AC2: A request from a user whose role lacks the required privilege is rejected with a 403 Forbidden response.
- AC3: Privileges are evaluated from the JWT claims at request time.

---

### L2-REQ-009: Current User Retrieval

**Traces to:** L1-REQ-002

The system shall allow authenticated users to retrieve their own account information.

**Acceptance Criteria:**

- AC1: An authenticated request to the current user endpoint returns the user's profile, roles, and active profile.
- AC2: Unauthenticated requests are rejected with 401 Unauthorized.

---

## Role and Privilege Management (L1-REQ-003)

### L2-REQ-010: Role CRUD Operations

**Traces to:** L1-REQ-003

The system shall support creating, reading, updating, and deleting roles.

**Acceptance Criteria:**

- AC1: A role can be created with a name.
- AC2: All roles can be listed.
- AC3: A role can be retrieved by its identifier.
- AC4: A role's name can be updated.
- AC5: A role can be deleted.

---

### L2-REQ-011: Privilege CRUD Operations

**Traces to:** L1-REQ-003

The system shall support creating, reading, updating, and deleting privileges associated with roles.

**Acceptance Criteria:**

- AC1: A privilege can be created specifying a role, an aggregate (resource name), and an access right (Read, Write, Create, Delete).
- AC2: Privileges can be listed, optionally filtered by aggregate.
- AC3: A privilege can be updated or deleted.

---

### L2-REQ-012: Predefined Roles

**Traces to:** L1-REQ-003

The system shall seed the following default roles on initial deployment: Member, Staff, BoardMember, SystemAdministrator, Support.

**Acceptance Criteria:**

- AC1: All five predefined roles exist after database seeding.
- AC2: Each predefined role has appropriate default privileges assigned.

---

### L2-REQ-013: Assign and Remove User Roles

**Traces to:** L1-REQ-003

The system shall allow assigning roles to and removing roles from users.

**Acceptance Criteria:**

- AC1: A role can be added to a user, granting the associated privileges.
- AC2: A role can be removed from a user, revoking the associated privileges.
- AC3: A user can hold multiple roles simultaneously.

## Profile Management (L1-REQ-004)

### L2-REQ-014: Create Profile

**Traces to:** L1-REQ-004

The system shall support creating profiles of type Member, BoardMember, StaffMember, or OnCall associated with a user.

**Acceptance Criteria:**

- AC1: A profile is created with a type, first name, last name, and phone number.
- AC2: Member profiles include an address (street, unit, city, province, postal code).
- AC3: BoardMember profiles include a board title.
- AC4: StaffMember profiles include a job title.
- AC5: The profile is associated with the specified user.
- AC6: If this is the user's first profile, it becomes both the current and default profile.

---

### L2-REQ-015: Update Profile

**Traces to:** L1-REQ-004

The system shall allow updating profile details.

**Acceptance Criteria:**

- AC1: First name, last name, phone number, and type-specific fields can be updated.
- AC2: The system returns the updated profile upon success.

---

### L2-REQ-016: Update Profile Avatar

**Traces to:** L1-REQ-004

The system shall allow setting or clearing a profile's avatar image.

**Acceptance Criteria:**

- AC1: A digital asset identifier can be set as the profile's avatar.
- AC2: The avatar can be cleared by setting it to null.

---

### L2-REQ-017: Delete Profile

**Traces to:** L1-REQ-004

The system shall allow deleting a profile.

**Acceptance Criteria:**

- AC1: The profile is removed from the system.
- AC2: The system returns confirmation of deletion.

---

### L2-REQ-018: Current and Default Profile Selection

**Traces to:** L1-REQ-004

The system shall track which profile is the user's current active profile and which is the default.

**Acceptance Criteria:**

- AC1: A user can switch their current profile among their associated profiles.
- AC2: The default profile is automatically set when the first profile is created.
- AC3: The current profile context determines the user's identity for messaging and maintenance requests.

---

### L2-REQ-019: Multiple Profiles Per User

**Traces to:** L1-REQ-004

The system shall allow a single user to have multiple profiles of different types.

**Acceptance Criteria:**

- AC1: A user can have profiles of type Member, BoardMember, StaffMember, and OnCall simultaneously.
- AC2: Each profile operates independently with its own identity context.

---

### L2-REQ-020: Board Member Management

**Traces to:** L1-REQ-004

The system shall provide specific management operations for board member profiles.

**Acceptance Criteria:**

- AC1: A board member's title, name, and avatar can be updated.
- AC2: A board member profile can be removed.

---

## Maintenance Request Workflow (L1-REQ-005)

### L2-REQ-021: Create Maintenance Request

**Traces to:** L1-REQ-005

The system shall allow members to submit maintenance requests.

**Acceptance Criteria:**

- AC1: A maintenance request is created with the requesting profile's identifier, address, phone number, description, and unattended unit entry preference.
- AC2: The initial status is set to "New".
- AC3: The creation date is recorded.
- AC4: A domain event is published upon creation.

---

### L2-REQ-022: Receive Maintenance Request

**Traces to:** L1-REQ-005

The system shall allow staff to receive (acknowledge) a maintenance request.

**Acceptance Criteria:**

- AC1: A request in "New" status can be transitioned to "Received".
- AC2: The receiving staff member's profile identifier and receipt date are recorded.
- AC3: A request not in "New" status cannot be received.

---

### L2-REQ-023: Start Maintenance Request

**Traces to:** L1-REQ-005

The system shall allow staff to mark a maintenance request as work started.

**Acceptance Criteria:**

- AC1: A request in "Received" status can be transitioned to "Started".
- AC2: The work start timestamp is recorded.
- AC3: Unit entry details are captured.
- AC4: A request not in "Received" status cannot be started.

---

### L2-REQ-024: Complete Maintenance Request

**Traces to:** L1-REQ-005

The system shall allow staff to mark a maintenance request as completed.

**Acceptance Criteria:**

- AC1: A request in "Started" status can be transitioned to "Completed".
- AC2: The work completion date and completing staff member's name are recorded.
- AC3: A request not in "Started" status cannot be completed.

---

### L2-REQ-025: Maintenance Request Status Enforcement

**Traces to:** L1-REQ-005

The system shall enforce a strict forward-only status progression: New -> Received -> Started -> Completed.

**Acceptance Criteria:**

- AC1: Status can only transition in the defined forward order.
- AC2: Backward transitions or skipping statuses are rejected.

---

### L2-REQ-026: Update Maintenance Request Description

**Traces to:** L1-REQ-005

The system shall allow updating the description of a maintenance request.

**Acceptance Criteria:**

- AC1: The description field can be updated.
- AC2: A domain event is published capturing the change.

---

### L2-REQ-027: Update Maintenance Request Work Details

**Traces to:** L1-REQ-005

The system shall allow updating work details on a maintenance request.

**Acceptance Criteria:**

- AC1: Work details describing the work performed can be added or modified.
- AC2: A domain event is published capturing the change.

---

### L2-REQ-028: Maintenance Request Comments

**Traces to:** L1-REQ-005

The system shall allow adding comments to a maintenance request.

**Acceptance Criteria:**

- AC1: A comment with a body and creator identifier is added to the request.
- AC2: The comment creation timestamp is recorded.
- AC3: Comments are persisted as owned entities of the maintenance request.

---

### L2-REQ-029: Maintenance Request Digital Assets

**Traces to:** L1-REQ-005

The system shall allow attaching digital assets (photos/files) to a maintenance request as evidence.

**Acceptance Criteria:**

- AC1: A digital asset can be associated with a maintenance request.
- AC2: Multiple digital assets can be attached to a single request.

---

### L2-REQ-030: Retrieve User's Maintenance Requests

**Traces to:** L1-REQ-005

The system shall allow users to view their own maintenance requests.

**Acceptance Criteria:**

- AC1: An authenticated user can retrieve only the maintenance requests they submitted.
- AC2: Results are filtered by the user's current profile identifier.

---

## Event Sourcing and Audit Trail (L1-REQ-011)

### L2-REQ-047: Domain Event Persistence

**Traces to:** L1-REQ-011

The system shall persist all domain events to an immutable event store.

**Acceptance Criteria:**

- AC1: Each stored event records the stream identifier (aggregate ID), event type, aggregate type, serialized event data, and creation timestamp.
- AC2: Events are versioned and sequenced per stream.
- AC3: A correlation identifier links related events.
- AC4: Events are stored before the SaveChanges transaction completes.

---

### L2-REQ-048: Event Publishing

**Traces to:** L1-REQ-011

The system shall publish domain events via a notification service after persistence.

**Acceptance Criteria:**

- AC1: Events are extracted from aggregate roots during the saving process.
- AC2: Events are published to subscribers after the database transaction succeeds.
- AC3: Aggregate root stored events are cleared after extraction.

---

### L2-REQ-049: Event Store Query

**Traces to:** L1-REQ-011

The system shall allow querying the event store for audit purposes.

**Acceptance Criteria:**

- AC1: An API endpoint exposes the event store for retrieval.
- AC2: Events can be queried by stream identifier and aggregate type.

## Modular Monolith Architecture (L1-REQ-012)

### L2-REQ-050: Module Decomposition

**Traces to:** L1-REQ-012

The system shall be organized into explicit modules within a single deployable backend, including Identity, Profile, Maintenance, Document, Asset, Messaging, and CMS-related capabilities.

**Acceptance Criteria:**

- AC1: Each module has a clearly defined responsibility and ownership boundary.
- AC2: All modules are deployed together as one backend application.
- AC3: Module boundaries are enforced through internal interfaces, application services, and domain abstractions rather than network calls.

---

### L2-REQ-051: In-Process Module Collaboration

**Traces to:** L1-REQ-012

The system shall coordinate cross-module workflows through in-process application services and domain notifications inside the modular monolith.

**Acceptance Criteria:**

- AC1: Cross-module workflows execute without service-to-service HTTP calls.
- AC2: Core workflows do not require an external message bus to complete.
- AC3: Internal events or notifications are dispatched within the same runtime and are observable through application logging and audit records.

---

### L2-REQ-052: Shared Data Platform

**Traces to:** L1-REQ-012

The system shall use a shared data platform while preserving module ownership of data structures.

**Acceptance Criteria:**

- AC1: The platform uses a single database deployment for the modular monolith.
- AC2: Tables or schemas are owned by specific modules and are not modified outside approved module contracts.
- AC3: Transactions can span multiple modules when required by a single business operation.

---

## Application Surfaces and API Layer (L1-REQ-013)

### L2-REQ-053: CMS-Driven Public Web App

**Traces to:** L1-REQ-013

The system shall provide a public-facing web app whose content and presentation are driven by CMS-managed data.

**Acceptance Criteria:**

- AC1: Public pages can render CMS-managed content blocks such as hero sections, board information, and cooperative announcements.
- AC2: Published themes and content updates can appear in the public web app without redeploying the application.
- AC3: Publicly available content can be retrieved through read-optimized endpoints intended for anonymous or low-friction access.
- AC4: Member-facing authenticated journeys, when required, reuse the same identity and authorization model as the rest of the platform.

---

### L2-REQ-054: Admin Backend

**Traces to:** L1-REQ-013

The system shall provide an authenticated admin backend for staff, board members, and administrators.

**Acceptance Criteria:**

- AC1: The admin backend supports management of users, roles, profiles, maintenance requests, documents, digital assets, themes, and CMS content.
- AC2: Admin capabilities are protected by role-based authorization.
- AC3: The admin backend supports editorial workflows for creating, updating, publishing, and retiring public-facing content.

---

### L2-REQ-055: Shared Backend API

**Traces to:** L1-REQ-013

The system shall expose a shared backend API consumed by both the public web app and the admin backend.

**Acceptance Criteria:**

- AC1: Both applications use the same modular monolith backend.
- AC2: Public content endpoints are separated from privileged admin operations through authorization and route design.
- AC3: The API exposes CMS content, published documents, and assets needed by the public web app while also supporting operational workflows for the admin backend.

---

### L2-REQ-056: RESTful Endpoint Design

**Traces to:** L1-REQ-013

The system shall expose all operations via RESTful HTTP endpoints following standard conventions.

**Acceptance Criteria:**

- AC1: POST is used for resource creation, GET for retrieval, PUT for updates, DELETE for removal.
- AC2: Successful creation returns HTTP 200/201.
- AC3: Resource not found returns HTTP 404.
- AC4: Validation errors return HTTP 400.
- AC5: Server errors return HTTP 500.

---

### L2-REQ-057: Pagination Support

**Traces to:** L1-REQ-013

The system shall support paginated retrieval of collections.

**Acceptance Criteria:**

- AC1: Paginated endpoints accept page size and page index parameters via URL path segments (e.g., `/page/{pageSize}/{index}`).
- AC2: Paginated endpoints are available for Users, Profiles, Maintenance Requests, Documents, and CMS-managed content.
- AC3: The response includes the requested page of results.

---

### L2-REQ-058: Authorization Enforcement

**Traces to:** L1-REQ-013

The system shall enforce authorization on all protected API endpoints.

**Acceptance Criteria:**

- AC1: All endpoints except explicitly public content, asset-serving, authentication, and invitation validation endpoints require a valid JWT token.
- AC2: Privileged endpoints are decorated with authorization attributes specifying required operations and aggregates.
- AC3: Unauthorized requests return HTTP 401; forbidden requests return HTTP 403.

---

### L2-REQ-059: Request Validation

**Traces to:** L1-REQ-013

The system shall validate all incoming requests before processing.

**Acceptance Criteria:**

- AC1: FluentValidation validators are defined for create and update operations.
- AC2: Validation is integrated into the MediatR pipeline via a behavior.
- AC3: Invalid requests are rejected before reaching the command/query handler.

---
